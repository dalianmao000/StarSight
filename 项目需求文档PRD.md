## 1. 文档概述

### 1.1 项目简介

数据服务平台面向企业级客户，提供基于 AI Agent 架构的**全域用户画像**、**智能标签挖掘**、**广告效果优化**、**金融风控建模**与**商业洞察分析**能力。平台以多智能体协同为核心，实现从"业务需求→数据需求→模型构建→交付输出"的全链路自动化。

### 1.2 术语定义

| 术语 | 定义 |
|:---|:---|
| **DMP** | Data Management Platform，数据管理平台，用于用户数据整合与精准营销 |
| **Lookalike** | 相似人群扩展，基于种子用户找到行为相似的潜在客户 |
| **MTA** | Multi-Touch Attribution，多触点归因，分析各渠道对转化的贡献 |
| **HITL** | Human-in-the-Loop，人在回路，关键节点需人工审批确认 |
| **PSI** | Population Stability Index，群体稳定性指标，用于监控模型漂移 |
| **KS** | Kolmogorov-Smirnov，风控模型区分度指标 |
| **LTV** | Lifetime Value，用户生命周期价值 |

### 1.3 目标读者

- 项目经理 / 产品经理
- 数据架构师 / 算法工程师
- 研发团队（后端 / 大数据 / AI）
- 测试团队
- 业务方 / 客户

### 2.2 M1：用户画像与标签体系

#### 2.2.1 功能列表

| 功能编号 | 功能名称 | 功能描述 | 优先级 |
|:---|:---|:---|:---|
| M1.1 | 标签定义与创建 | 支持可视化配置标签规则，含人口属性、行为轨迹、兴趣偏好、商业价值、风险等级五大维度 | P0 |
| M1.2 | 特征自动挖掘 | 基于历史数据自动挖掘候选特征，支持规则特征与 embedding 特征双模式 | P0 |
| M1.3 | 标签效果验证 | 提供 AB 测试机制，计算准确率 / 覆盖率 / PSI，支持灰度发布 | P0 |
| M1.4 | 标签血缘追踪 | 记录标签的数据来源、计算逻辑、更新时间，支持溯源与影响分析 | P1 |
| M1.5 | 标签生命周期管理 | 支持标签的创建→验证→发布→衰减→下线全生命周期 | P1 |
| M1.6 | 人群圈选与画像报告 | 支持多标签组合圈选人群，生成群体画像报告 | P1 |
| M1.7 | API 对外服务 | 提供标签查询、人群圈选、属性透视的 RESTful API | P2 |

#### 2.2.2 用户故事

```
作为：运营人员
我想要：通过可视化界面配置标签规则，无需 SQL 即可创建新标签
以便：快速响应业务需求，缩短标签上线周期

作为：数据分析师
我想要：查看标签的血缘关系和数据来源
以便：在标签出问题时有据可查，快速定位根因

作为：广告投放人员
我想要：基于多个标签组合圈选目标人群
以便：提升定向精准度，优化广告投放效果
```

#### 2.2.3 验收标准

- 标签准确率 ≥ 90%，核心人群覆盖率 ≥ 85%
- 标签创建到上线周期 ≤ 3 个工作日（含效果验证）
- 标签血缘可追溯至原始数据表和字段
- API 响应时间 ≤ 200ms（P99）

### 2.3 M2：广告 DMP 与效果优化

#### 2.3.1 功能列表

| 功能编号 | 功能名称 | 功能描述 | 优先级 |
|:---|:---|:---|:---|
| M2.1 | 人群圈选与 Lookalike | 支持标签组合圈选，基于种子用户扩展相似人群 | P0 |
| M2.2 | 多触点归因（MTA） | 计算各渠道对转化的贡献度，支持 Shapley / Markov 模型 | P0 |
| M2.3 | 投放策略模拟 | 基于历史 CTR/CVR 模拟不同出价 / 频次 / 预算下的 ROI | P1 |
| M2.4 | 实时竞价对接 | 对接 DSP/SSP API，支持人群包与策略参数实时下发 | P1 |
| M2.5 | 频次控制与预算分配 | 支持广告频次上限配置，多渠道预算智能分配 | P2 |
| M2.6 | 广告效果看板 | 展示 CTR/CVR/ROI/归因分析等核心指标，支持自定义维度和时间范围 | P2 |

#### 2.3.2 用户故事

```
作为：广告投放运营
我想要：上传一批高价值用户作为种子，找到相似的潜在客户
以便：扩大投放人群，提升转化率同时控制成本

作为：市场总监
我想要：了解各渠道对最终转化的贡献度
以便：合理分配营销预算，优化投入产出比

作为：广告优化师
我想要：模拟不同出价策略下的 ROI 预期
以便：在正式投放前评估方案可行性，降低试错成本
```

#### 2.3.3 验收标准

- Lookalike 扩展相似度 ≥ 85%（通过线上小流量验证）
- MTA 归因模型准确率 ≥ 80%（对比端到端转化数据）
- 策略模拟响应时间 ≤ 5 秒
- 支持至少 3 个主流 DSP 平台对接

### 2.4 M3：金融风控与信用评估

#### 2.4.1 功能列表

| 功能编号 | 功能名称 | 功能描述 | 优先级 |
|:---|:---|:---|:---|
| M3.1 | 反欺诈识别 | 基于资金 / 设备 / IP / 行为关系图构建，识别异常子图与欺诈团伙 | P0 |
| M3.2 | 信用评分卡 | 支持逻辑回归 / XGBoost 等模型，提供可解释性分析（SHAP/LIME） | P0 |
| M3.3 | 贷前准入 / 贷中监控 / 贷后预警 | 全流程风控，覆盖信贷业务各阶段 | P0 |
| M3.4 | 实时风险预警 | 基于 Flink CEP 实现实时行为监控，触发阈值告警与干预策略 | P1 |
| M3.5 | 模型公平性报告 | 检测模型在不同群体（如年龄、性别、地域）上的偏差 | P1 |
| M3.6 | 合规审计 | 记录全量决策日志，支持监管要求的版本溯源与数据脱敏证明 | P0 |

#### 2.4.2 用户故事

```
作为：风控经理
我想要：看到新申请用户的反欺诈评分和关键风险因子
以便：快速判断是否通过准入，降低坏账风险

作为：合规负责人
我想要：查看风控模型的公平性报告和决策解释
以便：满足监管检查要求，向审计机构提供证据

作为：贷后管理
我想要：实时监控借款人行为异常，自动触发预警
以便：在风险恶化前及时介入，最大限度减少损失
```

#### 2.4.3 验收标准

- 反欺诈模型 KS 值 ≥ 0.45
- 信用评分卡 AUC ≥ 0.72
- 实时预警延迟 ≤ 500ms
- 合规审计日志保留 ≥ 5 年，支持即时查询
- 所有风控决策可解释，支撑监管检查

## 3. 非功能需求

### 3.1 性能需求

| 指标 | 要求 |
|:---|:---|
| **API 响应时间** | P95 ≤ 200ms，P99 ≤ 500ms |
| **实时计算延迟** | 反欺诈 / 风控预警 ≤ 500ms |
| **批量处理吞吐量** | 日处理 PB 级数据，支持水平扩展 |
| **系统可用性** | SLA ≥ 99.5% |
| **NL2SQL 响应时间** | ≤ 3 秒 |

### 3.2 安全需求

| 需求类别 | 具体要求 |
|:---|:---|
| **数据安全** | 敏感数据加密存储（手机号、身份证等），传输全程 HTTPS |
| **权限控制** | 最小权限原则，RBAC + 行级 / 列级权限控制 |
| **审计追溯** | 全量操作日志，保留 ≥ 5 年，支持即时查询 |
| **合规要求** | 支持数据脱敏，满足《个人信息保护法》要求 |
| **金融合规** | 风控模块满足监管审计要求，支持模型公平性检测 |
| **部署模式** | 支持私有化 / 混合云部署，数据不出客户环境 |

### 3.3 可用性需求

| 需求 | 具体要求 |
|:---|:---|
| **故障恢复** | RTO ≤ 15 分钟，RPO ≤ 5 分钟 |
| **降级策略** | 单点故障不影响全局，关键操作有 fallback |
| **监控告警** | 系统异常 / 模型漂移 / 预算超限实时告警 |
| **多端支持** | 支持桌面端 Web、钉钉 / 飞书移动端 |

### 3.4 可维护性需求

| 需求 | 具体要求 |
|:---|:---|
| **版本管理** | 模型、标签、API 支持版本管理，支持灰度回滚 |
| **配置化** | 业务规则、审批流程、告警阈值支持可视化配置 |
| **可观测性** | 全链路 tracing，支持 Prometheus + Grafana |

## 4. 验收里程碑

| 阶段 | 时间 | 交付物 | 验收标准 |
|:---|:---|:---|:---|
| **M1-M2 基础搭建** | 第 1-2 月 | Agent 框架部署、ETL 流水线、NL2SQL 基础版、1 个核心标签自动挖掘流程 | 标签发现周期 ↓ 50%，SQL 生成准确率 ≥ 85% |

## 5. 风险与约束

| 风险类型 | 风险描述 | 缓解措施 |
|:---|:---|:---|
| **数据质量不足** | 现有数据分散、埋点缺失，影响标签和模型效果 | 数据治理先行，制定数据质量基线 |
| **业务需求模糊** | 业务方难以量化描述需求，导致返工 | 使用 5-Dimension 分析法结构化拆解需求 |
| **成本失控** | LLM 调用量大，Token 成本超预算 | 强制嵌入 Cost Controller，严格配额管理 |
| **监管合规风险** | 金融场景监管要求严格，合规评估周期长 | 合规前置设计，预留监管沟通时间 |
| **团队能力缺口** | 无 AI Agent 实战经验，学习曲线陡 | 分阶段引入，外援 + 内训结合 |
